从 Telegram 打开云手机链接很方便。

运营收到 QCCBot 链接，点开后进入云手机列表，再选择运行中的 Android 云手机。进入设备后，云手机里可能已经安装 Telegram，也可能正停在 Telegram 登录页面。

这个流程很好理解，但它带来一个重要问题：哪些内容应该留在云手机会话里，哪些内容可以在外部 Telegram 聊天里沟通？

## 一句话安全原则

外部 Telegram 聊天用来协同，QCCBot 云手机会话用来完成真正的移动端操作。

团队可以在 Telegram 里发 QCCBot 链接、云手机 ID、简单状态和交接说明。但验证码、账号资料、私聊内容和敏感截图，不应该发到群聊里。

## 风险通常出现在哪里

风险不在于从 Telegram 打开链接。

风险在于大家把不同环境里的信息混在一起：

- 云手机里出现 Telegram 登录页面；
- 有人在外部群聊里索要验证码；
- 截图里带着私聊预览；
- 运营把云手机里的账号备注复制到 Telegram 群；
- 队友分不清当前操作是在实体手机 Telegram，还是在云手机里的 Telegram；
- 团队不知道哪个云手机 ID 对应哪个业务。

只要边界清楚，这个流程就可以很安全。

## 哪些能发，哪些别发

可以把下面这张表当成实用规则。

| 信息 | 能不能发到外部 Telegram 聊天 | 更适合放在哪 |
| --- | --- | --- |
| QCCBot 登录链接 | 可以，发给正确团队 | Telegram 聊天 |
| 云手机 ID | 可以 | Telegram 交接或 QCCBot |
| 运行中、关机等设备状态 | 可以 | Telegram 摘要 |
| “Telegram 停在登录页”这类 App 状态 | 可以，前提是不含隐私 | Telegram 摘要 |
| 登录邮箱、验证码、密码 | 不要 | 在安全流程里处理 |
| 私聊内容 | 不要 | 云手机会话内查看 |
| 含个人信息的完整截图 | 通常不要 | 在 QCCBot 内复核 |
| 自动化日志 | 只发摘要 | QCCBot 任务上下文 |

这样 Telegram 仍然方便，但不会变成敏感数据的临时仓库。

## 更安全的登录交接

你给的截图里有一个典型情况：进入云手机后打开 Telegram，页面要求选择登录邮箱。

不安全的交接是：

> 把邮箱验证码发群里。

更安全的交接是：

> 云手机 ID 2059... 当前停在 Telegram 登录邮箱页面。请在云手机会话里完成登录，不要把验证码或账号恢复信息发到群聊。

这样既说明了当前状态，也保护了账号。

## 运营继续操作前先检查这些

在云手机里使用 Telegram 前，运营可以先问自己：

- 我现在是在 QCCBot 云手机里操作，还是在个人手机 Telegram 里？
- 当前云手机 ID 是哪一台？
- 这是不是正确的业务账号或流程？
- 屏幕上有没有私聊、验证码或账号资料？
- 下一步只是普通 App 操作，还是敏感账号动作？
- 这一步应该人工处理，还是可以交给脚本？

这些问题很基础，但能避免很多误操作。

## QCCBot 应该怎么描述才准确

QCCBot 应该被描述为云手机平台。

它让用户打开云手机列表、进入远程 Android 设备、按设备隔离不同业务，并在云手机环境里操作 App。如果配置了脚本或 AI 辅助流程，那也是 QCCBot 侧的能力。

Telegram 的角色要更具体：

- 可以是分享 QCCBot 链接的地方；
- 可以是团队沟通的地方；
- 可以是云手机里运行的 App；
- 可以承载简单交接信息。

这些已经足够，不需要夸大。

## 一套简单团队规则

可以先定这些规则：

- QCCBot 链接只发给正确团队；
- 交接时说云手机 ID，不说账号秘密；
- 验证码不要进群聊；
- 私密页面在云手机会话里复核；
- 用文字摘要代替敏感截图；
- 交接信息写清楚设备、App 状态和下一步安全动作。

如果团队需要从浏览器打开和管理多台 Android 云手机，[QCCBot 可以让运营更清楚地进入云手机，并把不同 App 工作流分开处理](https://www.qccbot.com/)。

## 最后确认一下

准确说法不是“Telegram 管理云手机”。

准确说法是：

> 运营可以从 Telegram 里的 QCCBot 链接进入控制台，在 QCCBot 里打开云手机，然后在云手机里的 Android 环境中使用 Telegram。

这个说法具体、可信，也和真实产品流程一致。

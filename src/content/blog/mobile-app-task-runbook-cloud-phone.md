如果一个移动 App 任务每周都要做，它就不应该只存在某个人脑子里。它需要一份运行手册。

运行手册不用很长，只要说明任务做什么、怎么跑、什么叫成功、会遇到哪些失败、谁负责处理异常。

## 简单答案

一份好的云手机任务运行手册，应该包含任务目的、设备分组、App 要求、脚本名称、成功标准、常见失败、AI 恢复规则、人工审核规则、日志位置和负责人。它不需要复杂，但要让其他同事也能接手。

## 为什么需要运行手册

没有运行手册，团队会依赖口头经验：

- 只有一个人知道用哪个脚本；
- 不知道某个设备分组为什么存在；
- 失败处理方式不一致；
- AI 恢复边界不清楚；
- 新同事上手很慢。

运行手册能让自动化流程可交接。

## 一个简单模板

可以按这个结构写：

| 模块 | 写什么 |
| --- | --- |
| 任务目的 | 这个任务解决什么问题 |
| 设备分组 | 哪些云手机运行 |
| App 状态 | 登录、权限、版本要求 |
| 脚本 | 脚本名称和最近测试时间 |
| 成功标准 | 什么算完成 |
| 常见失败 | 失败分类 |
| AI 恢复 | AI 可以处理什么 |
| 人工审核 | 什么必须停止 |
| 日志 | 在哪里看结果 |
| 负责人 | 谁维护流程 |

大部分运营任务用这张表就够了。

## 语言要简单

运行手册不是写给开发一个人看的。

不要只写：

“activity 切换后 selector error。”

可以写：

“脚本进入上传页后找不到上传按钮，检查 App 页面是否改版。”

简单语言能让团队更快行动。

## 失败后更新手册

真实失败后，最适合更新运行手册。

遇到这些情况就补充：

- 新弹窗出现；
- 脚本更新；
- AI 恢复规则变化；
- 某类失败变多；
- 任务负责人变化。

这样自动化不会越来越神秘。

## 常见问题

### 每个任务都要写很长文档吗？

不用。短而清楚的手册，比没人看的长文档更有用。

### 谁负责运行手册？

通常是负责业务结果的人，比如运营负责人。

### AI 恢复规则也要写进去吗？

要。不然团队不知道哪些失败可以自动恢复。
